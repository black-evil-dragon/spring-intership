
# Функциональные требования

## Авторизация

- Система должна поддерживать регистрацию нового пользователя
- Система должна поддерживать вход пользователя по логину и паролю
- Пароль должен храниться только в хэшированном виде
- После входа пользователь должен получать доступ в соответствии со своей ролью
- В системе должны быть роли: **`Администратор`**, **`Менеджер`**, **`Исполнитель`**
- **`Администратор`** должен иметь возможность создавать и управлять пользователями и ролями
- Доступ к защищенным операциям должен проверяться на сервере


## Управление задачами

1. Система должна позволять:
	- Создавать задачи
	- Просматривать список задач и карточку задачи
	- Редактировать задачи
	- Удалять задачи

2. У задачи должны быть:
	- Обязательные поля: **`Название`**, **`Описание`**
	- Дополнительные поля: **`Крайний срок`**, **`Приоритет`**
	- **`Постановщик`**
	- **`Исполнитель`**

- **`Менеджер`** и **`Админ`** должны иметь полный доступ к управлению задачами
- **`Исполнитель`** должен видеть только доступные ему задачи и свои назначения.

## Назначение пользователей

1. Система должна:
	- Позволять назначать **`Исполнителя`** на задачу
	- Хранить **`Постановщика`** задачи

- Один пользователь может быть **`Постановщиком`** многих задач
- Один пользователь может быть **`Исполнителем`** многих задач
- При изменении **`Исполнителя`** система должна сохранять актуальную привязку к задаче

# Нефункциональные требования

## Безопасность

- Пароли пользователей должны шифроваться
- Доступ к API должен защищаться через **Spring Security**
- Роли должны проверяться на уровне контроллеров или сервисов
- Неавторизованный запрос должен блокироваться системой
- Ошибки авторизации и аутентификации должны возвращаться в понятном формате


## Данные и производительность

- В качестве СУБД должна использоваться **PostgreSQL**
- Входящие данные должны валидироваться через **Hibernate Validator**
- Основные операции CRUD должны выполняться без избыточных запросов к базе
